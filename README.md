# 酒狐悄悄话增强版 v2.3.1 WineFox-Daily

> 一只可爱的酒狐女仆，随时随地为主人送上暖心悄悄话。

基于 [Koishi](https://koishi.chat/) 框架 + OneBot 协议的 QQ 机器人插件。当前主线已经完成 **v2.3 经济体系重设计**，并在 **v2.3.1** 补齐了 **偏好/群设置、渲染队列与缓存、渲染诊断/统计、玩家存档备份恢复（需指定管理员）** 等运维与稳定性能力。

## 安装

将 `WineFox-Daily` 文件夹放入 Koishi 项目的插件目录中，然后在 Koishi 控制台启用该插件。

```
你的Koishi项目/
├── plugins/
│   └── WineFox-Daily/
│       ├── index.js
│       ├── package.json
│       ├── runtime_config.js
│       ├── lib/
│       ├── data/
│       └── memory/
└── koishi.yml
```

### 关于 `koishi`

本插件在代码里只直接依赖 `koishi`，并以 `peerDependencies` 的形式声明。  
通常情况下，**不需要**在本插件目录里单独保留 `node_modules/` 或 `package-lock.json`，使用宿主 Koishi 项目的依赖即可。

### 关于图片功能

图片卡片依赖宿主环境已经安装并启用 Puppeteer 服务插件。  
WineFox 侧已声明 `puppeteer` 为可选注入服务；宿主若未启用 Puppeteer，插件会自动回退为文字输出。

## 经济与成长说明

### 双货币

- **好感度**：只负责角色关系成长和等级解锁
- **狐狐券**：通过签到、问答、猜拳、猜数字、开瓶、委托、被动触发、成就等方式获得，用于商店和酿酒消费

### 日常奖励节奏

当前主线已经按“玩家每天活跃约 10 分钟，也能拿到明显正反馈”做过一轮平衡调整：

- 高频玩法改为 **单次奖励更明显**
- 同时通过每日领取上限，避免单一玩法刷穿经济
- 例如：
  - `每日酒狐`：`+2` 狐狐券（每日 1 次）
  - `酒狐抽签`：`+2` 狐狐券（每日 1 次）
  - `酒狐故事`：单次 `+2` 狐狐券（每日限次）
  - `酒狐猜拳` 胜利：`+3` 狐狐券（每日限次）
  - `酒狐猜数` 胜利：`+4 ~ +6` 狐狐券（每日限次）
  - `酒狐问答` 答对：`+4` 狐狐券（每日限次）
  - `酒狐签到`：基础狐狐券奖励已提升

### 10级好感

v2.3 已将旧版 6 级好感扩展为 **10 级**，并细化成长节奏。  
好感变化后会显示：

- 当前好感值
- 下一等级名称
- 距离下一等级所需点数
- 下一等级解锁内容

如果当天已经达到好感上限，系统会明确提示“今日好感已达到上限”，不会再继续误导性地显示本次已经加了好感。

### 每日免费体验

每天会从低等级门槛指令中随机选出 4 个指令，允许玩家临时无视等级门槛使用。  
帮助菜单和群内首次公告会同步显示当日免费池。

## 性能与稳定性配置（可选）

> 这些配置不会改变玩法内容，主要用于 **减少 Puppeteer 负载**、提升高峰期稳定性、降低磁盘写入频率。  
> 推荐通过 `runtime_config.js` 覆盖（优先级高于 Koishi 面板配置）。

### 0) 运维指定管理员（敏感指令权限）

- `opsAdminIds`：允许执行敏感运维指令的 QQ / 用户ID 列表（建议填 QQ 号字符串）

敏感指令包括：

- `酒狐存档备份` / `酒狐存档恢复`
- `酒狐账本 @某人`（查询他人账本）

注意：

- 若 `opsAdminIds` 为空，上述敏感指令会直接拒绝执行（即使你有 Koishi 的 authority）
- 上述敏感指令的 `authority` 设为 `0`，权限**完全由** `opsAdminIds` 控制（避免 Koishi 权限配置差异影响）
- 普通的管理员指令（例如 `酒狐UI` / `酒狐渲染诊断`）仍然按 authority 控制

### 0.1) 玩家存档备份/恢复（仅 opsAdminIds）

WineFox 会在插件目录下创建备份目录：

- `WineFox-Daily/player_backups/`
  - 备份目录名：`backup-YYYYMMDD-HHMMSS`
  - 每个备份包含 `manifest.json`（记录备份时间、版本、包含文件等）

备份内容遵循“只包含玩家数据相关存档”的原则（不会备份季节循环/全局主题等全局状态）。当前包含的玩家数据存档大致包括：

- `memory/affinity.json`（好感）
- `memory/inventory.json`（背包）
- `memory/commission.json`（委托）
- `memory/favorites.json`（收藏）
- `memory/checkin.json`（签到）
- `memory/brewing.json`（酿酒）
- `memory/quiz.json`（问答）
- `memory/achievements.json`（成就）
- `memory/ticket-reward-ledger.json`（狐狐券领取次数账本）
- `memory/dynamic-shop.json`（动态商店每日限购记录）
- `memory/pending_submissions.json`（投稿队列）
- `memory/prefs.json`（个人/群偏好）

相关指令：

- `酒狐存档备份`：创建一次玩家存档备份
- `酒狐存档列表 [N]`：查看备份列表
- `酒狐存档清理 [N]`：只保留最近 N 个备份（`-n` 仅预览不删除）
- `酒狐存档恢复 <id> -f`：从备份恢复（危险，会覆盖现有存档；恢复后**请尽快重启插件/Koishi**）

注意：`player_backups/` 已加入 `.gitignore`，不会被提交到仓库。

### 1) 图片渲染并发队列

- `imageRenderMaxConcurrency`
  - `0`：不限制（保持现状）
  - `2~4`：推荐值（建议 `<= puppeteer.renderPoolSize`）
- `imageRenderQueueTimeout`
  - 排队超时后会自动回退为文字（若 `imageFallbackToText=true`）

如果你使用的是魔改版 Puppeteer（支持 `reusePage/renderPoolSize` 页面池），建议：

- Puppeteer：`reusePage=true`，`renderPoolSize=2~4`
- WineFox：`imageRenderMaxConcurrency <= renderPoolSize`

### 2) 图片渲染缓存

- `imageCacheEnabled`：默认关闭
- `imageCacheMaxEntries`：缓存上限（LRU 淘汰）
- `imageCacheDefaultTtlMs`：默认 TTL

缓存目前主要用于**低变化**指令（例如：帮助/分类/总数/故事目录等），能显著减少重复渲染带来的卡顿。

如需手动清理缓存（例如刚切主题/联调 UI），可使用管理员指令：

```text
酒狐缓存清理
```

注意：该指令清理的是 **WineFox-Daily 的图片渲染结果缓存（内存）**，不会删除 `memory/*.json` 业务存档。

### 2.1) 渲染诊断与统计

- `酒狐渲染诊断 [-i]`：输出渲染链路状态、队列/缓存状态、存档健康与“最近渲染统计摘要”
- `酒狐渲染统计 [N]`：查看最近 N 条渲染记录（含缓存命中、排队等待、渲染耗时、失败原因）
  - `酒狐渲染统计 -c`：清空统计（仅内存）

渲染统计是**仅内存**的环形缓冲区，最多保留条数可通过 `renderMetricsMaxEntries` 调整。

### 3) 高频存档写入防抖

- `ioDebounceMs`：默认 `0`（关闭）

开启后会对部分“历史记录类存档”（例如最近抽取历史、故事历史）做写入合并，减少高频 IO。  
建议值：`200~800`（按宿主磁盘性能调整）。

## 配置

当前 `exports.Config` 项数较多（包含图片开关、运维与性能开关等），其中图片相关开关已扩展到 30+ 项。  
配置加载优先级：

`runtime_config.js` > `runtime_config.json`（兼容旧版） > Koishi 后台配置 > Schema 默认值

如需关闭某个图片卡片，可直接在 `runtime_config.js` 中将对应 `imageXxx` 设为 `false`。

## 从旧版迁移

1. 将原来的 `酒狐文本存储foxqqh用.txt` 复制到 `WineFox-Daily/data/quotes.txt`
2. 在 Koishi 中禁用旧插件，启用新插件
3. 语录文件格式兼容，无需额外修改
